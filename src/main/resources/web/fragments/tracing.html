OpenTracing is a vendor-neutral API for distributed instrumentation and tracing which Coherence has adopted support for as of version 14.1.1.0. The OpenTracing feature can be dynamically enabled or disabled at the cluster or member level using JMX or Management over REST.

OpenTracing will be bootstrapped automatically by Coherence when the appropriate dependencies are on the classpath (see the Jaeger dependency within the demo's pom.xml). However, any compatible OpenTracing implementation may be used as long the library also provides a TracerResolver implementation (Jaeger happens to be an implementation that supports this). Configuration of Jaeger is typically performed by a series of environment variables (see the documentation here), which you can see used within the demo code for the Launcher, StartMemberResource, and StartSecondaryResource classes.

In order to capture tracing spans for JAX-RS, see the TracingFilter class. This filter intercepts in-coming requests and outbound responses and creates/updates spans for all JAX-RS resources defined in the ServiceResourceConfig class, however, the TracingFilter class can be used with any JAX-RS resource.

This demo also uses JPA to persist cache entries to an in-memory database in order to show tracing spans being captured by the demo's use of the the OpenTracing java-jdbc library which allows tracing of any JDBC driver. Review the demo's persistence.xml to see how this is enabled.

Review the Utilities class to see how we instrumented existing code to logically group demo operations with any spans that may be generated by the actions the code takes.

Lastly, navigate to http://localhost:16686 to see the captured spans in a visual format via Jaeger UI. When reviewing spans within the UI, you will be able to see JAX-RS, Coherence, and JDBC spans along with their metadata and timings.

There are two ways to configure tracing within the demo. The first, is to use the Configure Tracing from the main drop down menu which allows configuration of tracing globally or by the member's role within the cluster (in the case of the demo, we arbitrarily chose odd and even roles). It's important to remember that while there is a facility to configure tracing at a cluster level, the change itself is applied the member level, so members added to the cluster after em a tracing configuration change, will not have the same change applied. Secondly, you may enable/disable tracing on a per-member basis by using the Actions menu associated with each cluster member. As you experiment, you will see the dashboard provide feedback on the tracing status of each cluster member.
